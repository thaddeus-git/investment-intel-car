## 1. 产品概述

### 1.1 一句话描述

> 自动化监控 5 家美股竞品公司的 SEC  filing、财务数据和关键事件，生成可交互的 Web 看板。将人工 4-6 小时的季度竞品分析压缩到 30 分钟内。

### 1.2 要解决的问题

| 痛点 | 现状 | 目标 |
|------|------|------|
| 每次季报季花 4-6 小时翻 SEC 网站找数据 | 人工逐家查阅 | 自动采集 + 标准化 |
| 财务数据散落在不同公司的 10-K/10-Q 里，格式不统一 | 手动 copy/paste | 一行 API 获取 5 家全部历史数据 |
| 8-K 事件容易被漏掉 | 平时不主动监控 | 新 filing 自动识别 + 分级预警 |
| Earnings Call 内容太长，老板没时间看 | 1 小时音频 → 0 分钟阅读 | 结构化纪要 5 分钟读完 |
| 跨公司对比费劲 | Excel 手动拉表 | 看板叠图，一眼看出差距 |

### 1.3 MVP 范围边界

```
MVP 覆盖（本期）                    不在 MVP（二期+）
─────────────────────────────────  ─────────────────────────
✅ 5 家美股上市公司                    ❌ 港股（易鑫 2858.HK）
✅ SEC Filing 自动监控                 ❌ 非上市公司（懂车帝/大搜车/天天拍车）
✅ 标准化财务数据提取                   ❌ 已倒闭/退市竞品（Vroom/人人车）
✅ 8-K 事件分级预警                    ❌ 钉钉/飞书推送
✅ Earnings Call 纪要（基于文本）       ❌ ASR 音频转写
✅ 单页 Web 看板                       ❌ PPT/PDF 自动生成
                                     ❌ 行业宏观数据接入
```

## 2. 用户与场景

### 2.1 目标用户

| 角色 | 需求 | 使用频率 |
|------|------|---------|
| **战略/战投负责人**（老板） | 快速了解竞品最新动态和财务对比，30 秒内获取关键信号 | 每周 1-2 次 |
| **分析师**（你） | 替代手工翻 SEC 网站的工作，自动获取标准化数据用于深度分析 | 季报季每日，平时每周 |

### 2.2 核心场景

**场景 1：季报季监控**
> 现在是 4 月底，5 家竞品扎堆发财报。早上 9 点打开看板，一眼看到 Carvana 昨天发了 10-Q 和 8-K（Item 2.02 业绩发布），Revenue $6,432M、Net Income $250M。自动对比 Carmax 同期数据，毛利率差距一目了然。8-K 里标红了高管离职事件。点击"Carvana 2026Q1 Earnings Call 纪要"链接，5 分钟读完管理层对下半年的展望。

**场景 2：日常风险预警**
> 周三下午收到看板上的红点提醒：Carvana 有人提交了 Form 144（内部人减持），累计本周已有 3 笔。同时优信发了一份 8-K Item 5.02（CFO 离职）。两条信息都标注为"⚠️ 需关注"。

**场景 3：跨公司对比分析**
> 季度战略会上老板问"Carvana 和 Carmax 的毛利率趋势谁更好"。打开看板，选两家公司，叠加过去 8 个季度的 Gross Margin 趋势图，2 秒出结论。

## 3. 功能需求

### 3.1 模块总览

```
┌─────────────────────────────────────────────────────┐
│                     Web 看板                          │
│         ┌──────────┬──────────┬──────────┐          │
│         │ 时间线    │ 财务趋势  │ 事件预警  │          │
│         └──────────┴──────────┴──────────┘          │
├─────────────────────────────────────────────────────┤
│  模块 A          │ 模块 B       │ 模块 C │ 模块 D    │
│  Filing 监控     │ 财务提取     │ 8-K 预警│ EC 纪要  │
├─────────────────────────────────────────────────────┤
│              SEC EDGAR API 数据层                     │
│     submissions API │ companyfacts API │ 第三方源     │
└─────────────────────────────────────────────────────┘
```

### 3.2 模块 A：SEC Filing 监控

**输入**: 5 家公司 CIK  
**输出**: 新 filing 列表（时间、类型、标题、关键摘要）  
**数据源**: SEC submissions API  
**更新频率**: 每 6 小时轮询

**功能规格**:

| 字段 | 说明 | 示例 |
|------|------|------|
| 公司名 | 哪家竞品 | Carvana Co. |
| Filing 类型 | 10-K / 10-Q / 8-K / S-1 / 13G 等 | 10-Q |
| 提交日期 | 当地时间 | 2026-04-29 |
| Accession Number | 唯一标识 | 0001690820-26-000035 |
| 关键标签 | 自动分类 | 📊 季报 / ⚠️ 事件 / 👤 高管 / 📋 其他 |

**验收条件**:
- [ ] 能检测到 2026-04-29 的 Carvana 10-Q 并在 1 个轮询周期内展示
- [ ] 5 家公司 filing 无遗漏（与 SEC 网站人工核对）
- [ ] 分类准确率 > 90%（10-K/Q → 财报类，8-K → 事件类，3/4/5 → 内部人，SC 13G → 持仓变动）

### 3.3 模块 B：财务数据提取

**输入**: 5 家公司 CIK  
**输出**: 标准化财务指标（季度 + 年度）  
**数据源**: SEC companyfacts XBRL API（单次请求获取全部历史）  
**更新频率**: 检测到新 10-K/10-Q 时触发

**核心指标**:

| 指标类别 | 具体指标 | XBRL Tag（示例：Carvana） |
|----------|---------|---------------------------|
| **收入** | Total Revenue | `RevenueFromContractWithCustomerExcludingAssessedTax` |
| **盈利** | Gross Profit | `GrossProfit` |
|  | Operating Income | `OperatingIncomeLoss` |
|  | Net Income | `NetIncomeLoss` |
|  | EPS (Basic) | `EarningsPerShareBasic` |
| **成本** | Cost of Revenue | `CostOfRevenue` |
|  | SG&A | `SellingGeneralAndAdministrativeExpense` |
| **资产负债** | Total Assets | `Assets` |
|  | Cash & Equivalents | `CashAndCashEquivalentsAtCarryingValue` |
|  | Long Term Debt | `LongTermDebt` |
| **现金流** | Operating Cash Flow | `NetCashProvidedByUsedInOperatingActivities` |
|  | CapEx | `PaymentsToAcquirePropertyPlantAndEquipment` |
|  | Free Cash Flow | 计算值 = OCF - CapEx |

> **已通过技术验证**: 2026-06-16 实测，Carvana companyfacts API 返回 347 个 us-gaap 标签，上述指标全部可获取。Revenue 使用 `RevenueFromContractWithCustomerExcludingAssessedTax`（2026Q1 = $6,432M），**不是** `Revenues` 或 `SalesRevenueNet`（后者仅 2019 年前数据）。

**XBRL 标签映射规则**:

由于不同公司可能使用不同的 XBRL 标签来表达同一指标，需要建立映射表：

```
指标: Total Revenue
├── Carvana (CVNA) → RevenueFromContractWithCustomerExcludingAssessedTax
├── Carmax (KMX)   → 待验证，可能相同或使用 SalesRevenueNet
├── AutoNation (AN) → 待验证
├── 优信 (UXIN)     → 待验证（中概股可能用不同标签体系）
└── 汽车之家 (ATHM) → 待验证

策略: 先用 companyfacts API 下载每家公司的所有可用标签，
      与基准标签做 fuzzy match + 人工校验第一期，
      建立映射表后全自动化。
```

**验收条件**:
- [ ] 5 家公司的 Revenue、Gross Profit、Net Income 与最新 10-K/Q 人工核对误差 < 1%
- [ ] 至少提取 12 个核心指标 × 5 家公司，覆盖率 > 90%
- [ ] 季度数据可追溯到 2022Q1（至少 4 年）

### 3.4 模块 C：8-K 事件预警

**输入**: 8-K filing accession number  
**输出**: 结构化事件卡片  
**数据源**: SEC submissions API（Item 号） + 8-K 文本（LLM 摘要）  
**更新频率**: 检测到新 8-K 时触发

**事件分级规则**:

| 级别 | Item 号 | 含义 | 展示 |
|------|---------|------|------|
| 🔴 **严重** | 2.02 | 业绩发布（超预期/暴雷） | 红色高亮 |
|  | 5.02 | 高管/董事离职或任命 | 红色高亮 |
|  | 1.01 | 重大合同/合作终止 | 红色高亮 |
| 🟡 **关注** | 5.03 | 章程/制度修改 | 黄色 |
|  | 5.07 | 股东会投票结果 | 黄色 |
|  | 3.01 | 退市/转板通知 | 黄色 |
| 🟢 **信息** | 9.01 | 财报附件 | 绿色 |
|  | 7.01 | Regulation FD 披露 | 绿色 |
| ⚪ **其他** | 其他 Item | 未分类 | 灰色 |

**LLM 摘要 prompt 模板**（用于 8-K 正文）:
```
你是二手车行业的竞品分析师。以下是 {company_name} 的 8-K filing 正文（Item {item_number}）。

请用 2-3 句中文总结关键内容，并给出"重要性评级"（高/中/低）。
如果涉及数字（金额、百分比、日期），必须在摘要中保留。

摘要：
```

**验收条件**:
- [ ] 8-K Item 2.02（业绩发布）和 5.02（高管变动）零漏报
- [ ] LLM 摘要能被"不懂财务的人"读懂
- [ ] 重要性评级与人工判断一致率 > 80%

### 3.5 模块 D：Earnings Call 纪要

**输入**: 公司名 + 季度  
**输出**: 结构化纪要（管理层要点、Q&A 关键问答、风险提示）  
**数据源**: 
- 主源：8-K Item 2.02 文本（业绩新闻稿，已有文字）
- 辅源：Seeking Alpha / Fool.com Transcript（公开，网页抓取）
- MVP 不做：ASR 音频转写（音频获取 + 质量 + 成本风险大，移到二期）

**技术路线**:
```
1. 检测到 8-K Item 2.02 → 获取正文（业绩 Press Release）
2. LLM 提取：财务亮点 + 管理层引述 + Forward-looking statements
3. 搜索 Seeking Alpha "{Ticker} earnings call transcript {Quarter}"
4. 如 transcript 可用 → LLM 提取 Q&A 要点
5. 如 transcript 不可用 → 标注"N/A"，跳过
```

**纪要模板**:
```markdown
# {Company} {Quarter} Earnings Call 纪要

## 📊 财务亮点
- Revenue: $X.XB (YoY +/-X%)
- Net Income: $X.XB
- [其他关键指标]

## 🗣️ 管理层陈述要点
1. [战略方向/市场判断]
2. [运营数据/里程碑]
3. [下季度/全年指引]

## ❓ Q&A 关键问答
- Q: [问题]
  A: [回答摘要]

## ⚠️ 风险提示
- [从 Forward-looking statements 提取的关键风险]

---
*数据来源: SEC 8-K Item 2.02, Seeking Alpha Transcript*  
*生成时间: {timestamp}*
```

**验收条件**:
- [ ] 业绩新闻稿（8-K Item 2.02）的摘要 100% 覆盖
- [ ] 第三方 transcript 获取成功率 > 60%（并非所有公司都被 Seeking Alpha 覆盖）
- [ ] 纪要阅读时间 < 5 分钟（非财务背景人员可复述核心内容）
- [ ] 关键财务数字与 10-Q/K 一致（LLM 不会"编数字"）← 用模块 B 的数据做交叉校验

## 4. 技术架构

### 4.1 数据流

```
                     SEC EDGAR API
                    ┌───────────────┐
                    │ submissions   │ ← 每 6h 轮询新 filing
                    │ companyfacts  │ ← 检测到 10-K/Q 时拉取
                    │ filing text   │ ← 检测到 8-K 时拉取
                    └───┬───────────┘
                        │
                    ┌───▼───────────┐
                    │  Data Pipeline │
                    │  ├─ fetch      │ Python + requests
                    │  ├─ normalize  │ pandas
                    │  ├─ classify   │ rule-based + LLM
                    │  └─ store      │ SQLite / JSON files
                    └───┬───────────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   ┌────▼────┐   ┌─────▼─────┐   ┌────▼────┐
   │ LLM 引擎 │   │ 第三方源    │   │ 看板     │
   │ Claude/  │   │ Seeking   │   │Streamlit│
   │ GPT-4    │   │ Alpha     │   │         │
   │ 摘要+纪要 │   │ Transcript│   │ localhost│
   └─────────┘   └───────────┘   └─────────┘
```

### 4.2 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| 数据采集 | Python 3.9+ + `requests` | SEC API 纯 HTTP，无需 headless browser |
| 数据清洗 | `pandas` | 处理 XBRL JSON → 结构化 DataFrame |
| 本地存储 | SQLite | 零配置，单文件，"竞品监控"数据量极小 |
| LLM | Claude API / OpenAI API | 文本摘要 + 纪要生成 |
| 看板 | Streamlit | Python 原生，最快出可视化 MVP |
| 定时任务 | `schedule` 库（单进程） / cron | MVP 阶段不需要 Celery/Redis |
| 部署 | 本地运行 / 轻量云服务器 | MVP 阶段不需要容器化 |

### 4.3 SEC API 调用规范

```python
# SEC 要求：User-Agent 必须包含机构名和联系邮箱
HEADERS = {
    "User-Agent": "CompanyName/1.0 (your-email@company.com)",
    "Accept-Encoding": "gzip, deflate",
}

# 速率限制：10 requests/second
# 实践发现实际限制更严格，建议加 200ms delay
BASE_URLS = {
    "submissions": "https://data.sec.gov/submissions/CIK{pad10}.json",
    "companyfacts": "https://data.sec.gov/api/xbrl/companyfacts/CIK{pad10}.json",
    "filing": "https://www.sec.gov/Archives/edgar/data/{cik_no_pad}/{acc_clean}/{acc}-index.htm",
}
```

### 4.4 数据模型（SQLite）

```sql
-- 公司元数据
CREATE TABLE companies (
    id INTEGER PRIMARY KEY,
    ticker TEXT UNIQUE,
    cik TEXT UNIQUE,          -- 10-digit padded
    name TEXT,
    sic_code TEXT,
    sic_description TEXT,
    fiscal_year_end TEXT      -- '1231'
);

-- Filing 记录
CREATE TABLE filings (
    id INTEGER PRIMARY KEY,
    company_id INTEGER REFERENCES companies(id),
    accession_number TEXT UNIQUE,
    form_type TEXT,           -- '10-K', '10-Q', '8-K', etc.
    filing_date DATE,
    items TEXT,               -- 8-K items, comma-separated
    is_processed BOOLEAN DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 标准化财务数据
CREATE TABLE financials (
    id INTEGER PRIMARY KEY,
    company_id INTEGER REFERENCES companies(id),
    metric_name TEXT,         -- 'revenue', 'gross_profit', etc.
    metric_value REAL,
    unit TEXT,                -- 'USD', 'shares', etc.
    fiscal_year INTEGER,
    fiscal_quarter INTEGER,   -- 1-4, NULL for annual
    form_type TEXT,           -- '10-K' or '10-Q'
    filing_date DATE,
    frame TEXT,               -- e.g., 'CY2026Q1'
    UNIQUE(company_id, metric_name, fiscal_year, fiscal_quarter)
);

-- 事件
CREATE TABLE events (
    id INTEGER PRIMARY KEY,
    company_id INTEGER REFERENCES companies(id),
    filing_id INTEGER REFERENCES filings(id),
    event_type TEXT,          -- 'earnings_release', 'executive_change', etc.
    severity TEXT,            -- 'high', 'medium', 'low', 'info'
    summary_cn TEXT,          -- LLM 生成的中文摘要
    raw_text TEXT,            -- 原始 8-K 文本
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Earnings Call 纪要
CREATE TABLE earnings_call_notes (
    id INTEGER PRIMARY KEY,
    company_id INTEGER REFERENCES companies(id),
    fiscal_year INTEGER,
    fiscal_quarter INTEGER,
    source TEXT,              -- '8k_item_202', 'seeking_alpha', etc.
    highlights TEXT,          -- 财务亮点 JSON
    management_remarks TEXT,  -- 管理层陈述
    qa_summary TEXT,          -- Q&A 摘要
    risks TEXT,               -- 风险提示
    full_text_md TEXT,        -- 完整 Markdown 纪要
```

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| SEC API 限流变更严格 | 中 | 中 | 加本地缓存，历史数据不重复拉；filing 检测用 submissions API 增量 |
| XBRL 标签跨公司不统一 | **高** | 中 | 首期每家手工校验建立映射表；发现新标签自动告警人工确认 |
| LLM "幻觉"编造财务数字 | 中 | 高 | 财务数字用模块 B 的 XBRL 结构化数据，LLM 只做文本摘要；交叉校验 |
| Seeking Alpha 反爬 | 中 | 低 | Transcript 是辅助源，核心依赖 8-K 文本；失败时标记 N/A 不阻塞 |
| Carvana IR 页面结构变化 | 低 | 低 | MVP 不依赖 IR 页面爬取；SEC API 是唯一必需数据源 |
| Streamlit 性能不足（数据量大后） | 低 | 中 | 数据预计算 + 缓存；必要时迁移到 Gradio 或简单 React |
| 代码中的财务逻辑错误 | 中 | 高 | 首要指标（Revenue/NI/Gross Profit）与财报人工核对；单元测试覆盖计算逻辑 |

### 关键假设

1. **SEC EDGAR API 保持稳定** — 目前 API 已稳定运行多年，结构变化概率低
2. **5 家公司 XBRL 标签体系可用** — US GAAP 标准下主流标签一致，中概股需额外适配
3. **Seeking Alpha Transcript 可获取** — 作为辅助源，不阻塞核心流程
4. **所有功能在本地/单服务器运行** — MVP 不涉及分布式架构
