# 竞品情报监控系统 MVP · PRD v2.0

> **版本**: v2.0（基于开源组件调研后重写）  
> **日期**: 2026-06-16  
> **状态**: 待评审  
> **v1.0 → v2.0 核心变化**: 从"自建 SEC 数据管道"改为"以 edgartools 为底座 + 薄封装 + 自建看板"，MVP 开发时间从 2-3 天压缩到 4-6 小时。

---

## 0. 前置决策：不造轮子

### 0.1 调研结论

在动手写任何代码之前，我们系统搜索了 SEC EDGAR 相关的开源生态。核心结论：

> **`edgartools` 一个 Python 库已经覆盖了 PRD v1.0 中模块 A/B/C/D 约 80% 的工作量。**  
> MVP 的挑战不是"怎么从 SEC 拿数据"，而是"怎么把数据组织成老板看得懂的样子"。

### 0.2 造 vs 不造的分解

| PRD 模块 | 不自建，直接用 | 需要自建 |
|----------|-------------|---------|
| **A. Filing 监控** | `edgartools`: `Company("CVNA").get_filings().latest()` | — |
| **B. 财务数据提取** | `edgartools`: `Company("CVNA").get_financials()` 一行拿全部标准化指标 | XBRL 标签中文命名映射（业务层） |
| **C. 8-K 事件预警** | `edgartools`: `eightk.items` 返回结构化 Item 号<br>`edgar-parser[llm]`: 自动调 Claude/GPT-4o 提取 8-K 文本 | 事件分级规则（按业务定义"重要"）<br>中文摘要生成 |
| **D. Earnings Call 纪要** | `edgar-parser[llm]`: `earnings_8k()` 提取 8-K 财报新闻稿 | Seeking Alpha transcript 抓取<br>中文纪要模板 |
| **E. Web 看板** | — | **全部自建**（Streamlit），但数据层全是现成 DataFrame |

### 0.3 选型决策

| 组件 | 选型 | 理由 |
|------|------|------|
| **SEC 数据底座** | [edgartools](https://github.com/dgunning/edgartools) (MIT, 3k+ ⭐) | 最成熟的 SEC Python 库。覆盖 20+ form 类型、XBRL 标准化、内幕交易、13F。内置 MCP server 可供后续 Agent 化。 |
| **8-K 文本 LLM 提取** | [edgar-parser](https://github.com/henrysouchien/edgar-parser) `[llm]` extra | 专门针对 8-K 财报新闻稿做了 Claude/GPT-4o 提取，弥补 edgartools 在非结构化文本理解上的短板 |
| **LLM** | Claude API (Haiku 4.5 做摘要, Sonnet 4.6 做纪要) | Haiku 成本低速度快适合批量摘要，Sonnet 做需要推理的纪要 |
| **看板** | Streamlit | Python 原生，与 edgartools DataFrame 无缝衔接 |
| **存储** | SQLite | 轻量，单文件，数据量小 |
| **定时任务** | `schedule` 库 / cron | MVP 阶段不需要 Celery |
| **部署** | 本地 localhost | MVP 先本地演示，通过再考虑云部署 |

### 2.2 与 v1.0 的关键差异

| | v1.0（从零自建） | v2.0（edgartools 底座） |

### 3.7 insider_tracker.py — Form 4 / Form 144 内部人交易追踪（~150行）

**背景**：
- **Form 4**: 高管、董事、持股 >10% 股东在交易后 **2 个工作日**内申报。买卖方向直接反映内部人对公司前景的判断——增持 = 看多，减持 = 看空。
- **Form 144**: 内部人计划出售 ≥5,000 股或 ≥$50,000 的**事前通知**。单笔减持计划可能持续数月，信号比单笔 Form 4 更强。

**edgartools 已提供的**：
```python
from edgar import Company

c = Company("CVNA")

# Form 4 列表（filing 元数据 + 文本）
form4s = c.get_filings(form="4").latest(30)

# ⭐ edgartools 内置的内部人交易解析（结构化 DataFrame）
txns = c.insider_transactions()
# 列: insider_name, relationship (CEO/Director/10% Owner), 
#     transaction_type (Buy/Sell), shares, price_per_share, 
#     total_value, transaction_date

# Form 144 列表（e​​dgartools 返回原始文本，需自建解析）
form144s = c.get_filings(form="144").latest(10)
# 通过 f.text() 或 f.markdown() 获取内容，正则提取股数和预估金额
```

**需要自建的**：
1. **Form 144 文本解析** — edgartools 不给 Form 144 结构化数据，需正则提取：计划出售股数、预估总金额、出售人、经纪商
2. **交易聚合** — 按 `公司 × 月份` 汇总内部人买卖笔数和金额
3. **内部人情绪指标** — 加权算法（见下方）
4. **中文摘要 prompt**（见下方）

**内部人情绪指标设计**：

```
InsiderSentiment = Σ(买入金额_i × 角色权重_i) - Σ(卖出金额_j × 角色权重_j)

角色权重参考：
  CEO / CFO              2.0  （最高决策层，信号最强）
  10% Owner              1.5  （大股东，接近内幕知情）
  Director               1.0  （董事会视野）
  EVP / SVP / VP         0.8  （执行层）

结果归一化到 [-100, +100] 区间：
  > +30  → 🟢 看多
  -30 ~ +30 → 🟡 中性
  < -30  → 🔴 看空
```

**LLM 摘要 prompt**（Claude Haiku 4.5）：

```
你是二手车行业的竞品分析师。以下是 {company} ({ticker}) 内部人最近 3 个月的交易记录：

{transaction_table}

请用 2-3 句中文总结内部人动向：
1. 总体净买入还是净卖出？涉及金额多大？
2. 是否有 CEO/CFO 级别的关键人物交易？如有请点名。
3. 与上一期相比，趋势是转向看多还是看空？
```

### 3.8 institutional_tracker.py — 13F 机构持仓追踪（~120行）

**背景**：
- 管理资产 **>$1 亿**的机构投资人每季度必须提交 **13F-HR**
- 可以追踪大基金对竞品股票的持仓变动：建仓 / 加仓 / 减仓 / 清仓
- **滞后约 45 天**（季末后 45 天内提交），但仍是最重要的机构动向数据源

**edgartools 已提供的**：
```python
from edgar import Company, ThirteenF

c = Company("CVNA")

# 获取竞品自身的 13F 申报（如竞品本身是机构管理人——通常不是）
# ⚠️ 核心用法不是按竞品找 13F，而是按 13F 找"谁持有竞品"

# 方式一：获取某份 13F 的持仓明细
filings_13f = c.get_filings(form="13F-HR").latest(5)  # 最近 5 季度
for filing in filings_13f:
    # edgartools ThirteenF 类可解析持仓
    # 返回: 股票代码、持仓股数、市值、变动等
    pass

# 方式二：edgartools 的 Filing 类支持直接获取 13F 数据
filing_obj = filings_13f[0]
# ThirteenF(filing_obj) 返回结构化持仓
```

**需要自建的**：
1. **反向查询逻辑** — "哪些大机构持有竞品股票"（不是竞品提交了什么 13F，而是谁的 13F 里出现了竞品 ticker）
2. **跨季度对比** — 同一机构 Q1 → Q2 的持仓变化（股数增减、市值变化）
3. **机构动向信号**（见下方）
4. ⚠️ **edgartools 的 13F 反向查询能力待验证**——如果 API 不支持反向查（"谁持有了 CVNA"），需要用 SEC EDGAR Full-Text Search 或第三方数据源补充。MVP 可先从已知机构列表（如 Renaissance、BlackRock、Vanguard 等 Top 50 机构）的 13F 中检索竞品 ticker

**机构动向信号设计**：
```
InstitutionalSignal = (新增机构数 - 退出机构数) × 2 + (加仓机构数 - 减仓机构数) × 1

判定阈值：
  > +5  → 🟢 机构看多
  -5 ~ +5 → 🟡 中性
  < -5  → 🔴 机构看空
```

**数据采集策略**：
- 频率：每季度一次（13F 本身是季报），季末 + 50 天后触发
- 目标机构池：可从 NASDAQ.com 或 WhaleWisdom 获取竞品的主要机构持有人名单作为种子
- 增量判断：对比上一季度持仓 CSV，标记变动 >5% 的机构

### 3.9 中国竞品特殊处理

**关键事实**：
- 优信 (UXIN) 和汽车之家 (ATHM) 是在美上市的中国公司（ADR 结构）
- 中国籍高管和董事通常**不持有美国注册证券**，而是持有开曼群岛控股公司或 VIE 层面的权益
- 这些权益**不受 SEC Section 16 管辖**，因此中国公司内部人极少甚至从不提交 Form 4
- Form 144 同理——中国公司内部人在境外减持通常不通过美国经纪商，不会产生 Form 144 申报

**处理策略**：

| Form 类型 | 美国竞品 (CVNA/KMX/AN) | 中国竞品 (UXIN/ATHM) |
|-----------|------------------------|---------------------|
| Form 4 | ✅ 正常采集 + 情绪分析 | ⚠️ 采集但不期望有数据；如出现则是**极重要信号**（罕见事件） |
| Form 144 | ✅ 正常采集 + 识别减持计划 | ⚠️ 同上 |
| 13F | ✅ 正常采集 | ✅ **正常采集**——13F 是机构申报其持仓，竞品只是被持有的标的，不受竞品自身注册地影响 |

**Dashboard 展示处理**：

```
内部人情绪指标卡片：
  CVNA  [🟢 +45]  ← 正常显示
  KMX   [🟡  +8]  ← 正常显示
  AN    [🟡 -12]  ← 正常显示
  UXIN  [— 数据不适用]  ← 灰色，hover 提示"该竞品无 SEC Section 16 申报义务"
  ATHM  [— 数据不适用]  ← 同上

机构持仓变动卡片：
  所有 5 家正常显示  ← 13F 数据对所有竞品可用
```

**实现建议**：
- `config.py` 中新增加一个标志位 `has_section16_obligation: bool` 区分两类公司
- `insider_tracker.py` 采集时 skip 标志位为 False 的公司（不报错、不记录空数据为异常）
- Dashboard 对标志位为 False 的公司显示 `"不适用"` 而非 `"无数据"`（语义不同）

## 6. 风险与已知坑位

| 风险 | 缓解 |
|------|------|
| **edgartools API 可能不完全匹配需求** | 先用 Carvana 一家做完整性验证，如果某些指标缺失再考虑补充手写 |
| **优信/汽车之家 XBRL 标签可能不同** | 中概股有时用 IFRS 而非 US GAAP 标签体系，需要验证 |
| **Seeking Alpha 反爬** | Transcript 是辅助源，失败不影响核心流程 |
| **edgar-parser[llm] 可能对 8-K 中文场景不够** | 它提取英文结构化数据，中文摘要仍需我们自己调 LLM |
| **LLM 成本** | Haiku 4.5 极便宜（~$0.25/1M tokens），5 家公司日常监控月成本预估 < $5 |
| **edgartools 依赖项冲突** | 用 venv 隔离环境 |
| **Form 4/144 数据稀疏** | 内部人交易频率低（月均 0-5 笔），看板"无数据"是正常态，需在 UI 上做好空状态设计（不是 "Error" 而是 "本月无内部人交易"） |
| **13F 滞后 45 天** | 13F 数据本身有时滞，分析时需标注"数据截止 {季末日期}"，避免与实时股价对照产生误导 |
| **13F 反向查询能力未验证** | edgartools 是否支持"谁持有了 CVNA"的反向检索待验证；如不支持，MVP 回退方案：维护 Top 50 机构 CIK 列表，逐个拉取 13F 后 grep 竞品 ticker |
| **中国竞品内部人数据缺失不是 bug** | 见 §3.9——这是 SEC 法规范围差异，不是系统缺陷。Dashboard 上明确标注"不适用"而非"无数据" |

### 已知坑位记录（来自技术验证）

1. **Carvana CIK 是 0001690820**，不是 0001691090（后者是个人）
2. **Revenue 标签不统一** — `edgartools` 的 `get_revenue()` 封装了这种差异，不需要手写映射
3. **SEC API 限流** — `edgartools` 内置了 rate limiting + 缓存，不需要自己处理
4. **8-K Item 号在 filings 元数据里** — `edgartools` 的 `EightK.items` 直接返回

## 附录 A：为什么 edgartools 而不是其他方案

| 候选 | 不选的理由 |
|------|-----------|
| edgartools | ✅ **选它** |
| edgar-parser 单独用 | 覆盖范围窄（只有 10-K/Q/8-K），缺少内幕交易、13F、filing 监控 |
| sec-edgar-toolkit | 功能重叠但社区小很多，没有 MCP server |
| Quartr/Aiera/AlphaSense | 企业定制价格、英文产品、开箱不符合中文内部工具场景 |
| 完全自建 | 2-3 天 vs 4-6 小时的差别，且自建容易漏掉边界 case |
